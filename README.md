# Q-Verify

Q-Verify is an evaluation framework for one research question:

> **Does alignment improve reasoning stability, or mainly clean-input accuracy?**

The contribution is evaluation methodology, not a new training algorithm:
- matched-compute comparison (`Base`, `DPO`, `DPO+Reflexion`, `Best-of-N`),
- internal perturbation analysis (hidden-state noise),
- multi-seed statistical reporting (CI + permutation tests),
- mechanism analysis across layers and noise types.

## Design Principles

1. **Controlled variables**: fixed task split, decoding protocol, and compute budget.
2. **Factor separation**: internal noise effects are separated from external prompting effects.
3. **Statistical discipline**: no single-run claim; report uncertainty and significance.
4. **Reproducibility-first**: manuscript claims come from CSV artifacts generated by scripts.

## Dataset

- Quantum benchmark source (QASM): [MQT Bench](https://www.cda.cit.tum.de/mqtbench/)
- Local manifests:
  - `data/task_manifests/train.jsonl`
  - `data/task_manifests/val.jsonl`
  - `data/task_manifests/test.jsonl`
  - `data/task_manifests/all_tasks.jsonl`

Main claims use `test.jsonl`.  
`all_tasks.jsonl` is supplementary stress testing only.

## Execution Paths

There are two valid workflows.  
`train_real_dpo.py` is **not** an old command; it is required only when you want to refresh the adapter.

### Path A: Evaluate Existing Adapter (fastest, no retraining)

```bash
python experiments/build_task_manifest.py --task_root benchmarks/tasks --target_min_tasks 20
QVERIFY_DEVICE=cpu STABILITY_SAMPLES=20 RECOVERY_EPISODES=20 FAILURE_SAMPLES=20 ./run_core_evals.sh
QVERIFY_DEVICE=cpu MANIFEST_PATH=data/task_manifests/test.jsonl SEEDS=42,43,44 N_TASKS=80 MECH_LAYERS=8,12,16 ./run_paper_suite.sh
```

### Path B: Re-train DPO Then Evaluate (full paper path)

```bash
python experiments/generate_data.py --manifest_path data/task_manifests/train.jsonl --n_samples 1200
python src/alignment/preference_builder.py
python experiments/train_real_dpo.py --device cpu --epochs 10
python experiments/build_task_manifest.py --task_root benchmarks/tasks --target_min_tasks 20
QVERIFY_DEVICE=cpu MANIFEST_PATH=data/task_manifests/test.jsonl SEEDS=42,43,44 N_TASKS=80 MECH_LAYERS=8,12,16 \
./run_paper_suite.sh
```

### What You Ran (recommended timeline)

1. Build manifests and run `run_core_evals.sh` as a pilot sanity check.
2. Generate preferences and train DPO adapter.
3. Run `run_paper_suite.sh` for multi-seed statistics and mechanism scans.
4. If step `[4/4]` fails due to shell/env issues, rerun:

```bash
python experiments/evaluate_recovery.py --episodes 50 --noise 0.2 --device cpu --manifest_path data/task_manifests/test.jsonl
python experiments/evaluate_failure_modes.py --n_samples 50 --device cpu --manifest_path data/task_manifests/test.jsonl
```

### Important Shell Note

Avoid:

```bash
QVERIFY_DEVICE=cpu python ... --device "$QVERIFY_DEVICE"
```

In strict shells this can expand to empty and cause `Device string must not be empty`.  
Use literal `--device cpu`, or `export QVERIFY_DEVICE=cpu` before the command.

## Core Artifacts (for Paper/RP)

Primary CSV artifacts:
- `results/analysis/stability_stats_summary.csv`
- `results/analysis/stability_stats_significance.csv`
- `results/analysis/mechanism_summary.csv`
- `results/analysis/recovery_data.csv`
- `results/analysis/failure_modes_data.csv`

Use `stability_stats_*` + `mechanism_*` as primary evidence.  
Use `recovery_*` + `failure_modes*` as supporting diagnostics.

Primary figures (suggested):
- `results/analysis/stability_stats_validity.png` (main)
- `results/analysis/mechanism_q-verify_gaussian.png` (mechanism)
- `results/analysis/recovery_comparison.png` (optional diagnostic)
- `results/analysis/failure_modes.png` (optional diagnostic)

README and RP do **not** need identical figure sets:
- README: 1-2 representative figures + artifact paths.
- RP/Paper: only figures directly supporting claims under page limits.

## 2x2 Interaction Design (Planned Extension)

To decouple internal perturbation and external prompting:
1. Sys1 (no Reflexion) + noise
2. Sys2 (with Reflexion) + noise
3. Sys2 with noise on first attempt only
4. Sys2 with noise on both first and reflection attempts

Additional process metrics:
- Revision Trigger Rate
- Revision Success Rate
- Refusal-to-Revise Rate

## Long-Run Mechanism Jobs

`evaluate_mechanism.py` supports:
- incremental CSV append,
- resume (`--resume`),
- progress logs (`--log_file`).

```bash
python experiments/evaluate_mechanism.py \
  --manifest_path data/task_manifests/test.jsonl \
  --n_tasks 80 \
  --seeds 42,43,44 \
  --layers 8,12,16 \
  --noise_types gaussian,uniform,signflip,dropout \
  --noise_levels 0.1,0.2,0.5 \
  --device cpu \
  --resume \
  --log_file results/analysis/mechanism_progress.log
```

## Reproducibility and Git Policy

- `results/` is git-ignored by default to keep repository size stable.
- Main claims are reproducible from scripts and manifest files.
- For sharing results publicly, export a **curated** artifact bundle to `artifacts/paper/` (small CSV + key figures) and track that folder.

## Legacy

Deprecated or exploratory scripts were moved to:
- `experiments/legacy/`

Legacy outputs under `results/preliminary_plots/`, `results/generalization/`, `results/topology/`, and `results/analysis/experiment_*.png` should not be used as primary evidence.
